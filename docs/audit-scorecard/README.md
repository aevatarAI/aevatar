# 架构审计评分规范（Standard Scorecard Spec）

本文档定义 `docs/audit-scorecard` 目录下所有评分卡的统一口径，避免不同审计批次使用不同标准。

## 1. 目的与适用范围

1. 统一“整体分 + 分模块分”的评分方法。
2. 统一证据标准（必须可定位到代码/脚本/命令结果）。
3. 统一扣分与不扣分边界，避免把“当前阶段技术选型”误记为架构缺陷。

适用对象：

1. `aevatar.slnx` 全量审计。
2. 子系统/能力审计（Workflow、Maker、Foundation、CQRS 等）。

## 2. 基线口径（强制）

以下两项属于当前阶段基线，不作为扣分项：

1. **仅有 InMemory 实现**（含 Runtime provider、ReadModel store、state/event/manifest 存储）不扣分。  
   前提：文档明确其环境定位（dev/test 或当前阶段默认实现）并可替换。
2. **Actor Runtime 仅 Local 实现**（尚未实现分布式 Actor）不扣分。  
   前提：架构分层清晰、接口边界稳定，不存在把“未分布式化”误记为设计缺陷。

说明：

1. 分布式能力是后续演进目标，可作为“后续项/增量加分项”，不是当前扣分项。
2. 历史评分卡若与本规范冲突，以本规范为准并在新审计中修正。

## 3. 标准化评分模型（100 分）

### 3.1 维度与权重

| 维度 | 权重 | 核心检查点 |
|---|---:|---|
| 分层与依赖反转 | 20 | Domain/Application/Infrastructure/Host 边界、跨层依赖方向 |
| CQRS 与统一投影链路 | 20 | `Command -> Event`、`Query -> ReadModel`、单一 Projection Pipeline |
| Projection 编排与状态约束 | 20 | Actor 化编排、lease/session 句柄、中间层事实态约束 |
| 读写分离与会话语义 | 15 | 命令与查询职责分离、异步收敛语义清晰 |
| 命名语义与冗余清理 | 10 | 命名一致性、无无效壳层/重复抽象 |
| 可验证性（门禁/构建/测试） | 15 | guards/build/test 可复现通过 |

### 3.2 计分规则

1. 总分 = 六个维度得分之和（满分 100）。
2. 每个维度先给满分，再按“已落地问题”扣分。
3. 扣分必须绑定证据：`文件路径:行号` 或命令输出。
4. 规划中但未落地的问题，不扣分；仅列为“后续风险/建议”。

### 3.3 等级映射

| 分数区间 | 等级 |
|---|---|
| 95-100 | A+ |
| 90-94 | A |
| 85-89 | A- |
| 80-84 | B+ |
| 70-79 | B |
| <70 | C |

## 4. 扣分项与非扣分项清单

### 4.1 可扣分（需证据）

1. Host 承载业务编排或跨层反向依赖。
2. CQRS 双轨实现导致读写链路不一致。
3. 中间层维护 `actor/entity/run/session` 事实态映射（服务级内存字典等）。
4. 通过 `actorId -> context` 反查替代显式 lease/session 句柄。
5. 关键架构守卫、构建、测试无法稳定通过。

### 4.2 不扣分（本规范强制）

1. 仅 InMemory 实现（可替换、定位清晰）。
2. Actor 仅 Local 实现（分布式 Actor 未实现）。

## 5. 分模块评分规范

1. 每个模块独立给出 0-100 分与一句结论。
2. 模块评分必须可追溯到同一套维度，不得使用“隐含私有标准”。
3. 推荐模块：Foundation+Runtime、CQRS、Workflow、Maker、Host、Docs+Guards。
4. 整体分与分模块分允许不完全线性一致，但差异需写明原因。

## 6. 审计执行流程（标准步骤）

1. 执行验证命令并记录结果：
   `bash tools/ci/architecture_guards.sh`
   `dotnet build aevatar.slnx --nologo`
   `dotnet test aevatar.slnx --nologo`
2. 收集结构证据（关键项目依赖、核心编排路径、投影路径、API 边界）。
3. 按第 3 节模型打分并生成“整体 + 分模块”结果。
4. 列出扣分项（按影响度排序）与改进建议（P1/P2）。

## 7. 评分卡输出模板（建议）

1. 审计范围与方法
2. 客观验证结果（命令 + 结果）
3. 整体评分（含维度表）
4. 分模块评分
5. 关键证据（加分项）
6. 主要扣分项（必须有证据）
7. 改进优先级建议

## 8. 版本与生效

1. 本规范自 `2026-02-20` 生效。
2. 新增评分卡必须遵循本规范。
3. 若规范更新，需在本文件追加版本记录并说明变更原因。
