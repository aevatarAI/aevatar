# Teams 首页信息结构

## 目标

`Teams` 首页应该在几秒内回答四个问题：

1. 当前正在查看哪个 `Scope`
2. 这个 `Scope` 下已经形成了哪些团队入口
3. 哪些团队运行正常，哪些需要处理
4. 用户下一步最应该点哪里

## 信息结构

首页按以下顺序组织：

1. 页面头
   - 标题：`我的 AI 团队`
   - 辅助信息：当前 `Scope`
   - 主动作：`组建新团队`

2. Scope 上下文
   - 只作为背景上下文出现
   - 允许切换 `Scope`
   - 不能把 `Scope ID` 当成团队名主展示

3. 概览指标
   - 团队入口数
   - 运行正常数
   - 需要处理数
   - 指标必须真实表达团队状态，不能使用技术性伪计数

4. 团队入口列表
   - 首页主内容
   - 每张卡只表达一个团队入口
   - 每张卡只展示：
     - 团队名
     - 状态
     - 默认入口
     - 最近更新时间
     - 关联服务
     - 主操作：`查看团队`

5. 次级提示区
   - 放运行信号缺失、草稿尚未形成入口、服务信息不可见等提醒
   - 这些提示是环境信息，不能和团队主卡混成一层

## 字段语义边界

- `Scope`：查看上下文，不是团队名
- `Team`：首页主角，是用户理解的业务对象
- `Binding Display Name`：入口或绑定名称，不天然等于团队名
- `Runtime Status`：运行信号，不等于团队生命周期
- `Service ID`：内部实现标识，不应该直接主展示

## 首页不该主展示的内容

- 原始 `Scope ID / UUID` 作为“当前 Team”
- `当前可见团队` 这类基于技术实现的伪计数
- 单卡场景下的卡片/列表切换
- `Not configured` 直接作为团队标题
- `default` 直接作为主服务主展示
- 顶部 warning 与卡片内部重复表达同一件事
- 无新增价值的重复蓝色标签

## 改造原则

- `Scope` 降级为上下文，不抢首页主位
- `Team List` 成为页面中心
- `Runtime / Service / Binding` 只作为团队状态的支撑证据
- 草稿、告警、信号缺失都放在次级层，不和团队主卡抢主视觉
- 单卡场景优先清晰，不为了未来假设保留多余控件
