# Studio Workflow Draft Phase 2 Alignment

## Scope

This document defines phase 2 for issue `#253` on top of phase 1 branch `fix/2026-04-20_workflow-draft-semantic-rename`.

Phase 1 has already aligned the storage and service names with draft semantics:

- `IWorkflowDraftStore`
- `WorkflowDraft`
- `ChronoStorageWorkflowDraftStore`
- `AppScopedWorkflowService.SaveDraftAsync(...)`

Phase 2 finishes the semantic alignment in three places that phase 1 intentionally left unchanged:

- workspace HTTP paths still say `workflows` even though they are draft-only
- `WorkflowFileResponse` still mixes draft-only and committed-only fields
- Studio save and publish semantics are still easy to misread from the API shape

## Locked Decision

### Decision 1: save draft and publish stay as two explicit operations

This is the only decision in this document, and it is fixed.

- `Save` persists editor draft state only
- `Publish` remains a separate scope-binding action
- saving a draft does **not** call `IScopeWorkflowCommandPort.UpsertAsync(...)`
- no automatic publish is added to the draft save path
- no new workspace-side publish endpoint is introduced

Everything else in this document is the implementation design that follows from that decision.

## Goals

- make draft APIs read as draft APIs
- make committed runtime APIs read as committed runtime APIs
- remove mixed-semantics DTOs
- keep Studio save honest about what it guarantees
- keep the existing explicit publish flow through scope binding

## Non-goals

- no auto-publish on save
- no compatibility alias for `/api/workspace/workflows`
- no change to scope binding semantics
- no query-time replay or projection priming
- no second publish transport under `/api/workspace`

## Semantic Model

### Draft

A draft is Studio-owned authoring state.

It may live in:

- a local workspace directory
- a scoped `IWorkflowDraftStore`

It includes:

- `workflowId`
- human-facing `name`
- file or draft addressing data
- raw YAML
- Studio layout
- draft update time

It does not claim runtime visibility.

### Committed workflow

A committed workflow is scope-visible runtime state.

It includes:

- committed identity
- source YAML when available
- parsed workflow document
- validation findings or fallback diagnostics
- committed update time

It does not expose workspace-only draft addressing fields.

### Publish

Publish means: bind the current workflow YAML bundle to a scope through the existing scope-binding flow.

The publish action remains:

- user-visible
- explicit
- separate from draft persistence

## API Design

### Draft routes

Replace all workspace workflow draft routes with draft-specific names:

- `POST /api/workspace/workflow-drafts`
- `GET /api/workspace/workflow-drafts`
- `GET /api/workspace/workflow-drafts/{workflowId}`
- `PUT /api/workspace/workflow-drafts/{workflowId}`
- `DELETE /api/workspace/workflow-drafts/{workflowId}`

These routes are draft-only whether the backing store is:

- local filesystem via `WorkspaceService`
- scoped draft store via `AppScopedWorkflowService`

### Committed runtime routes

Committed runtime reads stay on the scope APIs:

- `GET /api/scopes/{scopeId}/workflows`
- `GET /api/scopes/{scopeId}/workflows/{workflowId}`

Publish stays on the existing scope-binding endpoint:

- `PUT /api/scopes/{scopeId}/binding`

Committed workflow detail and list reads are not abstract in phase 2. The concrete surface is:

- backend: the existing scope workflow list/detail endpoints
- frontend: the existing `scopesApi.listWorkflows(scopeId)` and `scopesApi.getWorkflowDetail(scopeId, workflowId)` paths

Phase 2 updates those scope workflow endpoints to return committed-specific DTOs instead of continuing to share `WorkflowFileResponse` or any draft-side contract.

### Save route shape

The draft save path is split into create and update:

- `POST /api/workspace/workflow-drafts` for first save
- `PUT /api/workspace/workflow-drafts/{workflowId}` for subsequent saves

Request body:

```csharp
public sealed record SaveWorkflowDraftRequest(
    string DirectoryId,
    string WorkflowName,
    string? FileName,
    string Yaml,
    WorkflowLayoutDocument? Layout = null);
```

Notes:

- `WorkflowId` is removed from the body in both create and update
- `workflowId` is generated by backend on create and returned in `WorkflowDraftResponse`
- `workflowId` is stable across subsequent saves
- the same create/update semantics are used for both local workspace drafts and scoped drafts

### Workflow ID rule

Studio must treat `workflowId` as the stable identity of a draft, not as a display field.

Recommended rule:

- if the draft already has `workflowId`, reuse it
- if this is the first save, backend generates the initial `workflowId`
- if `DirectoryId` or `FileName` changes, that is a path move/rename of the same draft, not a new draft

Detailed rule set:

1. first save:
   - frontend calls `POST /api/workspace/workflow-drafts`
   - backend creates the initial `workflowId`
   - backend returns the generated `workflowId` in `WorkflowDraftResponse`
2. subsequent save:
   - frontend calls `PUT /api/workspace/workflow-drafts/{workflowId}`
   - backend trusts the path `workflowId` as the draft identity
   - backend may change `Name`, `DirectoryId`, `FileName`, `Yaml`, and `Layout` without changing `workflowId`
3. rename or move:
   - changing `FileName`
   - changing `DirectoryId`
   - or changing both
   all mean "move the same draft", not "create a new draft"
4. collision:
   - if the target path already belongs to a different `workflowId`, save fails with a conflict-style validation error
   - phase 2 must not silently overwrite another draft during move/rename

Phase 2 keeps the existing name-normalization semantics, but it decouples draft identity from local file path.

### Draft error contract

Phase 2 makes draft-save conflict behavior explicit.

#### HTTP statuses

- `400 Bad Request`
  - malformed request body
  - missing `DirectoryId`
  - empty `WorkflowName`
  - empty `Yaml`
  - invalid `workflowId`
  - unknown `DirectoryId`
- `404 Not Found`
  - `GET /api/workspace/workflow-drafts/{workflowId}` when the draft does not exist
  - `DELETE /api/workspace/workflow-drafts/{workflowId}` when the draft does not exist
  - `PUT /api/workspace/workflow-drafts/{workflowId}` when the draft does not exist
- `409 Conflict`
  - save attempts to move or rename a draft onto a path already occupied by a different draft
- `401/403`
  - unchanged from current scoped workspace resolution rules
- `502`
  - unchanged for backend adapter failures on scoped runtime-backed flows

#### Conflict payload

Path-collision saves return:

```json
{
  "code": "WORKFLOW_DRAFT_PATH_CONFLICT",
  "message": "Draft 'workflow-a' cannot move to 'team-a/review.yaml' because that path is already used by draft 'workflow-b'."
}
```

Contract rules:

- `code` is stable and machine-readable
- `message` is user-readable and can be surfaced directly in Studio notices
- phase 2 must not return a generic `400` for path-collision saves

Recommended backend implementation:

- introduce a draft-conflict exception path in application/service code
- map it to `409 Conflict` in `WorkspaceController`
- for scoped flows, use the same payload shape through `AppApiErrors.CreatePayload(...)`

### Local workspace move semantics

For local workspace drafts, phase 2 changes the storage rule from:

- `workflowId = base64(filePath)`

to:

- `workflowId = stable draft identity`
- `filePath = mutable storage location`

Required backend behavior for local workspace save:

1. load the existing draft by `workflowId`
2. compute the requested target path from `DirectoryId + FileName`
3. if no existing draft exists, create a new file at the target path
4. if the existing draft already uses the target path, update in place
5. if the target path differs:
   - write the file to the new target path
   - delete the old file for the same `workflowId`
   - preserve the same `workflowId`
6. return the new `FileName`, `FilePath`, `DirectoryId`, and `DirectoryLabel` in `WorkflowDraftResponse`

This is explicitly a move/rename of one draft.

Path ownership check:

- before writing the target path, backend must query whether another stored local draft already resolves to that path
- if another draft owns the path, backend returns the `409 WORKFLOW_DRAFT_PATH_CONFLICT` contract above
- if the same draft already owns the path, save proceeds normally

### Scoped draft semantics

For scoped drafts backed by `IWorkflowDraftStore`, `workflowId` is already the stable key.

Phase 2 keeps that model:

- changing `WorkflowName` updates draft content
- no path migration is needed
- synthetic `FileName` and `FilePath` returned to the UI remain projections of the stable `workflowId`

## DTO Design

### Draft response

Replace draft-side `WorkflowFileResponse` with:

```csharp
public sealed record WorkflowDraftResponse(
    string WorkflowId,
    string Name,
    string FileName,
    string FilePath,
    string DirectoryId,
    string DirectoryLabel,
    string Yaml,
    WorkflowLayoutDocument? Layout,
    DateTimeOffset UpdatedAtUtc);
```

Fields intentionally omitted:

- `Document`
- `Findings`

Reason:

- both are derived views of YAML, not authoritative draft facts
- current clients already parse YAML client-side in several flows
- keeping them in the draft DTO is what preserves the mixed-semantics problem

### Committed response

Replace runtime-side `WorkflowFileResponse` usage with:

```csharp
public sealed record WorkflowCommittedResponse(
    string WorkflowId,
    string Name,
    string Yaml,
    WorkflowDocument? Document,
    IReadOnlyList<ValidationFinding> Findings,
    DateTimeOffset? UpdatedAtUtc = null);
```

Fields intentionally omitted:

- `FileName`
- `FilePath`
- `DirectoryId`
- `DirectoryLabel`
- `Layout`

Reason:

- those fields describe draft storage, not committed runtime state

### Summary DTOs

Phase 2 splits list DTOs the same way it splits detail DTOs:

- `WorkflowDraftSummary`
- `WorkflowCommittedSummary`

`WorkflowSummary` is removed.

Implementation rule:

- workspace draft list endpoints return `WorkflowDraftSummary`
- scope runtime list endpoints return `WorkflowCommittedSummary`
- no generic summary type remains if it mixes draft and runtime semantics

Recommended shapes:

```csharp
public sealed record WorkflowDraftSummary(
    string WorkflowId,
    string Name,
    string Description,
    string FileName,
    string FilePath,
    string DirectoryId,
    string DirectoryLabel,
    int StepCount,
    bool HasLayout,
    DateTimeOffset UpdatedAtUtc);

public sealed record WorkflowCommittedSummary(
    string WorkflowId,
    string Name,
    string Description,
    int StepCount,
    DateTimeOffset? UpdatedAtUtc = null);
```

## Backend Responsibilities

### `WorkspaceController`

`WorkspaceController` becomes the host for draft routes only.

Required changes:

- rename route attributes from `workflows` to `workflow-drafts`
- rename action methods to draft-specific names
- add create route `POST("workflow-drafts")`
- change update save to `PUT("workflow-drafts/{workflowId}")`
- return `WorkflowDraftResponse`
- remove any code path where workspace draft endpoints return committed runtime DTOs

### `WorkspaceService`

`WorkspaceService` stays the local workspace draft implementation.

Rename surface:

- `ListWorkflowsAsync` -> `ListDraftsAsync`
- `GetWorkflowAsync` -> `GetDraftAsync`
- `SaveWorkflowAsync` -> `SaveDraftAsync`
- `ToWorkflowFileResponse(...)` -> `ToWorkflowDraftResponse(...)`

Behavior:

- local save still writes file content and layout
- YAML may still be parsed internally for summary derivation
- draft detail responses no longer expose parse products
- local save must support "same draft, new path" semantics

Phase 2 requires `WorkspaceService` to stop deriving draft identity from `filePath`.

Required service behavior:

- create draft and return backend-generated `workflowId`
- resolve existing draft by `workflowId`
- compute requested target path separately
- detect path collision with a different draft
- move file when target path changed
- preserve `workflowId` across move/rename

### `AppScopedWorkflowService`

`AppScopedWorkflowService` is the riskiest area because it still mixes:

- scoped draft reads
- scoped draft saves
- committed runtime reads

Phase 2 should split that shape logically even if it does not introduce a new class in the first patch.

At minimum, the service must expose two distinct internal mapping paths:

1. draft path
   - `ListDraftsAsync`
   - `GetDraftAsync`
   - `SaveDraftAsync`
   - `DeleteDraftAsync`
   - draft DTO mapping only
2. committed path
   - committed runtime read mapping only

Hard rule:

- `SaveDraftAsync(...)` writes only `IWorkflowDraftStore` plus persisted layout
- it must not call `IScopeWorkflowCommandPort.UpsertAsync(...)`

### `IStudioWorkspaceStore`

Phase 2 requires `IStudioWorkspaceStore` to support stable draft identity independent of file path.

The interface does not need to keep its current exact method set. It does need to support these operations:

- list drafts with stable `workflowId`
- get draft by `workflowId`
- save draft while changing file path for the same `workflowId`
- detect whether a target path already belongs to another `workflowId`
- delete draft by stable `workflowId`

Minimum acceptable implementation strategies:

1. extend `StoredWorkflowFile` persistence so `workflowId` is stored independently of `filePath`
2. maintain a workspace draft index that maps `workflowId -> current filePath`

The chosen implementation must also support:

- reverse lookup from target `filePath -> workflowId`
- conflict detection before write
- atomic enough move semantics that a successful save never leaves both old and new files as active owners of the same draft

Phase 2 should not preserve the current implicit `workflowId == encoded filePath` rule for local workspace drafts, because that makes same-draft move semantics impossible.

### Runtime query surface

Committed runtime reads should stay attached to existing scope APIs.

Phase 2 should prefer using the already-correct runtime surface instead of reintroducing runtime behavior through workspace draft endpoints.

### Studio models

Replace generic `StudioWorkflowFile` in [models.ts](/Users/liyingpei/Desktop/Code/aevatar/apps/aevatar-console-web/src/shared/studio/models.ts) with draft-specific types.

Recommended types:

- `StudioCommittedWorkflow`

Draft type must not contain:

- `document`
- `findings`

Those values should come from parse and validate flows.

### Editor load/save flow

Studio draft load path:

1. list drafts
2. fetch selected draft
3. parse YAML locally if the graph or structured document is needed

Studio draft save path:

1. serialize graph to YAML
2. if this is the first save, call `POST /api/workspace/workflow-drafts`
3. otherwise call `PUT /api/workspace/workflow-drafts/{workflowId}`
4. update local editor state from returned `WorkflowDraftResponse`, including backend-generated `workflowId`
5. run parse/validate locally as needed

### Publish flow

Publish remains exactly one user action:

1. build YAML bundle from current draft plus referenced draft workflows
2. call `studioApi.bindScopeWorkflow(...)`
3. show publish-specific success or failure notices

UI copy guidance:

- save notice: `Draft saved`
- publish notice: `Published to scope ...` or `Updated scope binding ...`

The UI should stop implying that save affects runtime visibility.

## Risks

### Risk 1: frontend still depends on draft parse products

Current Studio code uses `document` from fetched workflow payloads in some flows. Once draft DTOs stop carrying `document`, those paths must explicitly parse YAML after fetch.

### Risk 2: route rename misses mirrored clients

The repository has at least two frontend API layers that still reference workspace workflow routes. Phase 2 must update both the console web client and CLI-hosted frontend mirrors.

### Risk 3: service mixing survives under new names

If `AppScopedWorkflowService` keeps both draft and committed mapping interleaved, phase 2 may pass tests while preserving the conceptual tangle. Review should explicitly check responsibility boundaries, not just compile success.
