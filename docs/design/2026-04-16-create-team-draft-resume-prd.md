# Create Team 草稿恢复 PRD

## 1. 背景

当前 `Create Team -> Open Studio` 已经具备“新建团队模式”的基础语义，但还缺一个关键闭环：

1. 用户在 Studio 里点击了 `保存草稿`
2. 系统确实保存了行为定义草稿
3. 但用户回到 `Create Team` 后，看不到“这次创建流程对应的已保存草稿”
4. 下次再进入 Studio，也不能明确恢复到这份草稿

结果是用户只能知道“保存成功了”，但不知道：

1. 草稿现在和哪次创建团队流程绑定
2. 应该从哪里继续编辑
3. 再次点 `Open Studio` 是继续上次，还是又新开一份

这会让 `保存草稿` 的价值大打折扣。

## 2. 问题定义

当前系统里其实存在两种不同语义的“草稿”：

1. 行为定义草稿
   Studio 保存到 workspace/scope 的 workflow document
2. 创建团队草稿
   `teamName / entryName / 当前这次创建流程对应的已保存行为定义`

目前第 1 种草稿已经能保存；第 2 种草稿没有正式建模，只是部分信息停留在 URL 中。

因此会出现两个具体问题：

1. `Create Team` 页面缺少“已保存草稿”的可见入口
2. `Open Studio` 无法稳定表达“继续编辑当前创建流程的已保存草稿”

## 3. 目标

本轮目标是补齐“创建团队草稿恢复”闭环，但不新增独立后端表。

用户应该能够：

1. 在 Studio 里点击 `保存草稿`
2. 返回 `Create Team` 后，看见“当前创建流程已关联的草稿”
3. 再次点击 `Open Studio` 时，继续进入这份草稿
4. 明确区分“继续已有草稿”和“删除草稿”

## 4. 非目标

本轮不做：

1. 独立的 team draft 后端存储
2. 多草稿管理器
3. 跨设备同步“创建团队上下文”
4. 额外的团队目录页或草稿列表页

## 5. 核心产品判断

本轮采用“最小正确模型”：

1. `保存草稿` 继续只保存行为定义
2. 但创建团队流程要额外维护一个“草稿恢复指针”
3. 这个恢复指针属于创建团队流程本身，不等于“当前正在浏览的 workflow”

恢复指针包含：

1. `teamName`
2. `entryName`
3. `teamDraftWorkflowId`
4. `teamDraftWorkflowName`

其中：

1. `teamDraftWorkflowId` 只在成功点击 `保存草稿` 后更新
2. 用户在 Studio 里临时点开别的行为定义，不应污染这个指针

## 6. 状态模型

### 6.1 Create Team 页面状态

`Create Team` 页面负责持有：

1. 团队名称
2. 入口名称
3. 当前创建流程关联的已保存草稿 ID
4. 当前创建流程关联的已保存草稿名称

### 6.2 Studio 新建团队模式状态

Studio 新建团队模式负责持有：

1. 当前正在编辑的行为定义
2. 团队名称
3. 入口名称
4. 当前创建流程的草稿恢复指针

这两组状态不能混淆：

1. “当前正在编辑的行为定义”可以变化
2. “草稿恢复指针”只能在保存成功后变化

## 7. 目标用户路径

### 7.1 首次创建

1. 用户进入 `Create Team`
2. 输入团队名称与入口名称
3. 点击 `Open Studio`
4. 进入一个新草稿
5. 点击 `保存草稿`
6. 返回 `Create Team`
7. 页面显示“已保存草稿：xxx”
8. 再次点击 `Open Studio`
9. 直接恢复到这份草稿

### 7.2 中断后继续

1. 用户此前已经保存过草稿
2. 再次回到 `Create Team`
3. 页面直接显示已保存草稿摘要
4. 用户点击 `继续编辑`
5. Studio 打开这份草稿，而不是新建空白草稿

### 7.3 删除草稿

1. 用户当前已有已保存草稿
2. 决定删除这次创建流程的草稿
3. 在 `Create Team` 页面点击 `Delete Draft`
4. 系统真正删除当前创建流程关联的 workflow 草稿
5. 页面清空草稿恢复指针
6. 再点 `Open Studio` 时，进入新的空白草稿

## 8. 交互设计

### 8.1 Create Team 页面新增模块

在表单区下方增加一个轻量的“当前草稿”摘要卡。

有已保存草稿时展示：

1. 标题：`已保存草稿`
2. 草稿名称：`{teamDraftWorkflowName}`
3. 辅助说明：`这份草稿会作为当前团队创建流程的继续编辑入口`
4. 主动作：`继续编辑草稿`
5. 次动作：`Delete Draft`

没有已保存草稿时，不展示这张卡。

### 8.2 Open Studio 语义

`Open Studio` 的行为改成：

1. 若存在 `teamDraftWorkflowId`，则恢复该草稿
2. 若不存在，则新建空白草稿

按钮文案可以保持 `Open Studio`，但如果已有草稿，页面内应额外强调“会继续上次草稿”。

### 8.3 Save Draft 语义

在新建团队模式下：

1. `保存草稿` 成功后
2. 更新“草稿恢复指针”
3. 允许用户返回创建页继续稍后处理

## 9. 路由与数据传递

本轮不新增后端 team draft API，先通过现有路由显式传递恢复指针。

### 9.1 Create Team 路由参数

新增：

1. `teamDraftWorkflowId`
2. `teamDraftWorkflowName`

语义分别是：

1. 当前创建流程关联的已保存草稿 ID
2. 当前创建流程关联的已保存草稿名称

### 9.2 Studio 路由参数

新增：

1. `teamDraftWorkflowId`
2. `teamDraftWorkflowName`

说明：

1. `workflow` 仍表示“当前打开的行为定义”
2. `teamDraftWorkflowId` 表示“当前创建流程的恢复指针”
3. 这两个字段不能混用

## 10. 规则

### 10.1 何时更新恢复指针

只允许在以下时机更新：

1. 新建团队模式下点击 `保存草稿` 成功后

### 10.2 何时不能更新恢复指针

以下行为都不应更新恢复指针：

1. 只是点击左侧行为定义列表切换了当前定义
2. 只是从已有行为复制但还没保存
3. 只是修改了入口名称
4. 只是测试运行

## 11. 验收标准

### 11.1 可见性

1. 用户在新建团队模式下保存草稿后，返回 `Create Team` 能看到“已保存草稿”摘要
2. 摘要中能看到草稿名称

### 11.2 可恢复性

1. 用户从 `Create Team` 再次进入 Studio 时，能恢复到刚才保存的草稿
2. 不会错误创建一份新的空白草稿

### 11.3 语义正确性

1. 切换当前行为定义不会污染创建流程的草稿恢复指针
2. `entryName` 与 `workflowName` 仍保持各自独立语义

### 11.4 可删除性

1. 用户可以主动删除当前创建流程关联的已保存草稿
2. 删除后再次进入 Studio 会新建空白草稿

## 12. 实施建议

本轮建议前端一次完成：

1. 路由扩展
2. `Create Team` 页面草稿摘要
3. Studio 保存成功后的恢复指针更新
4. 回到 `Create Team` 的链接改造
5. 相关单测补齐

## 13. 前后端分工备注

当前前端只能正确表达“继续编辑已保存草稿”，还不能实现真实删除。

原因是仓库当前不存在删除 workflow 的正式后端接口与存储删除链路。

因此在后端接口补齐前，前端策略应为：

1. 展示 `Delete Draft` 按钮
2. 但按钮保持禁用
3. 明确说明：等待后端删除 workflow 接口后再接真删除

后端补齐后，再把该按钮接成真实删除。

后续若需要跨设备或长期保存，再把“创建团队草稿”升级为正式后端模型。
