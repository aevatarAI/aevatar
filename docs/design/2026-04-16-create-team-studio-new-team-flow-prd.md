# Create Team -> Studio 新团队流程 PRD

## 1. 背景

当前 `Create Team` 页面里的 `Open Studio`，实际打开的是一个通用 `workflow` 新草稿编辑器，而不是“新建团队”专属流程。

这导致用户在“我要创建一个新团队”的心智下，进入了一个混合了以下语义的页面：

- 当前空白草稿
- 当前 scope 的既有 binding
- 当前 scope 的既有 service
- 当前 scope 下的历史 workflow 列表

结果是用户很难判断：

- 我现在是在创建新团队，还是在编辑老团队
- `保存` 会保存什么
- `发布 / 更新团队入口` 会影响谁
- `draft / test03 / default` 分别是什么

## 2. 问题定义

当前体验的核心问题不是单个按钮文案，而是“新建团队流程”和“通用 Studio 编辑器”被混成了一个入口。

具体问题如下：

1. `Create Team -> Open Studio` 打开的不是“新团队创建流程”，而是“新 workflow 草稿”。
2. 新建团队场景下仍然展示当前 scope 的已发布团队入口信息，容易让用户误以为自己正在编辑现有团队。
3. 页面同时展示 `draft`、既有入口名、`serviceId`，但没有解释三者边界。
4. 顶部标题、副标题、底部发布模块来自不同事实源，用户会把它们误认为同一个对象。
5. `返回团队` 会返回既有团队详情，而不是 `Create Team`。
6. `保存` 与 `发布团队入口` 的语义区分不清。
7. 新建团队场景仍突出展示既有 workflow 列表，削弱“从零创建”的体验。
8. `draft` 作为默认标题直接暴露给用户，强化了错误心智。
9. `Create Team` 页面承诺“完成后进入 Team Details”，但当前流程里没有一个明确的“团队已创建”时刻。
10. `Create Team` 页面右上角和表单区内部同时出现 `Open Studio`，但两者行为完全一致，没有提供额外价值，反而破坏主次层级。
11. 顶部 4 张 summary card（`Studio / 行为 + 脚本 / Team Details / 新增后端流 0`）更像实现说明或工程注释，而不是用户创建团队时真正关心的信息。
12. `行为定义 / 脚本行为 / Agent 角色 / 集成` 这组 tag 在创建页里只是能力清单，无法帮助用户做当前决策，属于噪音。
13. `Saved Draft` 被直接放在“创建新团队”的同一页面和同一表单语境里，造成“继续旧草稿”和“开始新团队”两条路径混在一起。
14. 当页面已经存在 `Saved Draft` 时，用户再填写一个新的团队名称和入口名称，`Open Studio` 仍可能继续旧草稿，形成“新表单 + 旧草稿”混合语义。
15. 当前模型隐含“同一时刻只允许一个全局 saved draft”的假设，导致用户无法自然地在已有未完成草稿的情况下开始创建另一个团队。
16. 右侧 `已填写团队名称：xxx` 只是弱状态回显，占用版面但不提供决策价值。
17. `Delete Draft` 在当前后端能力未就绪时被禁用展示，会把一个未完成能力暴露到主链路，增加认知负担。

## 3. 目标

`Create Team` 的目标应该是：让用户以最小理解成本完成“从无到有创建一个新团队入口”。

这条流程必须满足：

1. 用户清楚知道自己在创建“新团队”，不是在编辑旧团队。
2. 用户清楚区分“团队名”“入口名”“草稿名”“serviceId”。
3. 在用户点击发布前，当前操作不会影响任何既有团队入口。
4. 发布动作完成后，系统给出明确成功反馈，并进入新团队详情页。
5. 创建新团队与继续旧草稿必须是两条显式路径，不能由页面静默替用户决定。
6. 页面在任一时刻只表达一个主任务，不同时混放“创建新团队”和“恢复旧草稿”两个主语。

## 4. 非目标

本轮不解决以下问题：

1. 多团队目录或 team catalog 的后端建模。
2. Scope 与 Team 的长期产品映射争议。
3. Studio 整体视觉重设计。
4. Workflow 编辑器本身的节点编辑能力。

## 5. 核心产品判断

1. `Create Team` 应该是一个“团队创建向导入口”，不是 Studio 的别名入口。
2. Studio 在这条路径里应该进入“新建团队模式”，而不是普通编辑模式。
3. 新建团队模式下，页面主语应是“待创建的团队”，不是“当前 scope 的已有 binding”。
4. 任何会影响既有团队入口的真实状态，都应该在未发布前降级或隐藏。
5. `Create Team` 页面只应保留一个主 CTA，不应出现两个同义的 `Open Studio`。
6. “继续旧草稿”必须是显式恢复动作，而不是默认覆盖“创建新团队”动作。
7. 未真正可用的 destructive action 不应出现在主流程首屏。

## 6. 用户故事

### 6.1 主故事

作为一个想创建新团队的用户：

1. 我点击 `Create Team`
2. 我输入团队的基本信息
3. 我选择从空白开始，或者从已有行为复制
4. 我进入 Studio 完成入口定义
5. 我保存草稿
6. 我确认发布
7. 我进入刚创建完成的团队详情页

### 6.2 安全预期

作为一个担心误改现网的用户：

1. 在点击发布前，我要明确知道当前修改不会影响已有团队
2. 如果当前 scope 已经有默认入口，我也不该被误导成“正在更新它”

## 7. 信息架构

新建团队流程建议拆成两层。

### 7.1 第 1 层：Create Team 向导

页面只负责收集最少必要信息：

1. 团队名称
2. 入口名称
3. 当前任务选择

当前任务只保留两个主选项：

1. 创建新团队
2. 继续之前的草稿

如果用户选择 `创建新团队`，页面才展示团队名称、入口名称与进入 Studio 的主动作。

如果用户选择 `继续之前的草稿`，页面只展示草稿摘要与进入 Studio 的恢复动作。

### 7.2 第 2 层：Studio 新建团队模式

进入 Studio 后，页面主语应切换成：

1. 正在创建的新团队
2. 当前团队入口草稿
3. 发布前不会影响已有团队

不应再以“当前 scope 的既有团队”作为页面主语。

## 8. 页面与状态设计

### 8.1 Create Team 页面

页面应包含：

1. 标题：`创建新团队`
2. 简短说明：`先定义团队名称和入口方式，再进入 Studio 完成细节`
3. 模式切换：`创建新团队` / `继续之前的草稿`
4. 表单区：团队名、入口名
5. 主动作：仅保留一个 `进入 Studio`
6. 次动作：`取消`

以下元素不应继续保留在页面首屏：

1. 顶部 4 张 summary card
2. 重复的右上角 `Open Studio`
3. 能力清单式 tag（`行为定义 / 脚本行为 / Agent 角色 / 集成`）
4. 没有明确动作语义的右侧弱状态文案

如果存在 saved draft，推荐改为单独的恢复区块，而不是和“创建新团队”表单上下直连。

恢复区块应至少包含：

1. 草稿名称
2. 关联的团队名称
3. 关联的入口名称
4. 最近更新时间
5. 主动作：`继续草稿`

### 8.2 Studio 新建团队模式

顶部必须明确表达当前状态：

1. 主标题：`正在创建团队「{teamName}」`
2. 副标题：`当前入口草稿：{entryName}`
3. 状态提示：`尚未发布，不会影响已有团队`

右侧主动作建议改为：

1. `保存草稿`
2. `测试运行`
3. `发布团队入口`

### 8.3 发布确认

发布前展示一个轻量确认层：

1. 团队名称
2. 入口名称
3. 来源：空白 / 复制自某个行为
4. 发布目标 scope
5. 影响说明：`发布后会创建新的团队入口`

### 8.4 发布成功

发布成功后要明确告诉用户：

1. 新团队已创建
2. 默认入口名称
3. 已跳转至 Team Details

### 8.5 已有 Saved Draft 时的正确行为

当系统发现用户已有 saved draft 时，不应默认把它绑定到当前表单。

正确行为应为：

1. 页面先提示：`你有一个未完成草稿`
2. 提供两个显式选择：
   `继续草稿`
   `创建新团队`
3. 只有在用户主动选择 `继续草稿` 后，`Open Studio` 才恢复旧草稿
4. 如果用户选择 `创建新团队`，则该次 `Open Studio` 必须进入新的创建链路，不能静默复用旧 draft

## 9. 新建团队模式下必须隐藏或降级的内容

以下内容在“新建团队模式”下不应以主信息出现：

1. 当前 scope 的既有默认入口摘要
2. `默认入口指向 xxx`
3. `更新团队入口`
4. 既有团队运行状态摘要
5. 将当前草稿与既有 service 强绑定的文案

以下内容可以保留，但必须降级为二级操作：

1. 查看已有 workflow 列表
2. 从已有行为复制
3. 查看既有团队详情
4. 删除草稿

## 10. 字段语义边界

以下字段必须在 UI 上明确分开：

1. 团队名称
   表达用户理解的业务对象名称

2. 入口名称
   表达首页和 Team Details 中展示的入口名称

3. workflow 草稿名
   表达当前编辑中的定义名称

4. serviceId
   仅是内部实现标识，不应在新建团队主链路里主展示

## 11. 推荐默认文案

### 11.1 顶部文案

1. `正在创建团队`
2. `尚未发布`
3. `发布前不会影响已有团队`

### 11.2 按钮文案

1. `进入 Studio`
2. `保存草稿`
3. `发布团队入口`
4. `从已有行为复制`
5. `返回创建页`
6. `继续草稿`
7. `创建新团队`

### 11.3 空状态文案

1. `当前还没有步骤`
2. `先从空白开始，或复制一个已有行为`
3. `发布后这里会成为新团队的默认入口`

## 12. 建议用户路径

### 12.1 从空白创建

1. 用户点击 `Create Team`
2. 如果存在 saved draft，先明确选择 `创建新团队`
3. 输入团队名称
4. 输入入口名称
5. 点击 `进入 Studio`
6. 在 Studio 新建团队模式中编辑入口
7. 点击 `保存草稿`
8. 点击 `发布团队入口`
9. 跳转到新团队详情页

### 12.2 从已有行为复制

1. 用户点击 `Create Team`
2. 如果存在 saved draft，先明确选择 `创建新团队`
3. 输入团队名称
4. 输入入口名称
5. 选择 `从已有行为复制`
6. 选择来源行为
7. 点击 `进入 Studio`
8. 在 Studio 新建团队模式中继续编辑
9. 点击 `发布团队入口`
10. 跳转到新团队详情页

### 12.3 恢复旧草稿

1. 用户点击 `Create Team`
2. 页面发现已有 saved draft
3. 页面单独展示草稿摘要
4. 用户点击 `继续草稿`
5. 进入 Studio 恢复对应草稿
6. 继续保存或发布

## 13. 验收标准

### 13.1 心智正确性

1. 用户不会把“新建团队”误解为“编辑现有团队”
2. 用户能明确知道“保存草稿”不会影响已发布入口
3. 用户能明确知道“发布团队入口”才会创建或落地团队入口
4. 用户不会因为已有 saved draft 而误把“创建另一个 team”操作继续到了旧 draft 上

### 13.2 可观察行为

1. `Create Team` 页面首屏只有一个主 CTA
2. 从 `Create Team` 进入 Studio 时，不展示既有团队作为当前主语
3. 顶部标题不出现 `draft` 作为最终用户主标题
4. 新建团队模式下不主展示 `serviceId`
5. `返回` 返回创建流程，不返回既有团队详情
6. 发布成功后跳转到新团队详情页，而不是停留在混合编辑态
7. 顶部 summary card 不再出现
8. 保存的旧草稿不会在用户填写新团队表单时被自动复用

## 14. 开放问题

1. 团队名称与入口名称是否允许默认相同，并在高级模式下再拆开
2. 从已有行为复制时，是否复制 YAML 即可，还是需要显式复制 layout / role / connector 引用
3. 发布成功后，Team Details 是否需要首屏展示“刚刚创建成功”的引导卡片

## 15. 一句话结论

`Create Team` 不能再只是“打开一个通用 Studio 草稿”。它必须成为一个明确、安全、可闭环的新团队创建流程。
