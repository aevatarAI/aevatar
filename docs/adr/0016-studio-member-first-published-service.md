# ADR-0016: Studio Member-First Published Service Identity

## Context

Studio is already intended to behave like a member lifecycle workbench:

`Create member -> Build -> Bind -> Invoke -> Observe`

But the current backend and frontend contract still leak a mostly scope-first model:

- `PUT /api/scopes/{scopeId}/binding` falls back to the scope default service when `ServiceId` is omitted
- Studio does not yet have a first-class backend `member` entity as the identity anchor
- `serviceId` still leaks into places where the product really means `memberId`
- the current `Create member` surface is implemented too narrowly and does not yet match the intended `workflow / script / gagent` creation model

That leaves Studio with the wrong primary object:

- `workflow file`
- `script definition`
- `published service`
- `binding revision`

instead of:

- `member`

At the same time, the desired product model is already clear in the current Studio design work:

- `member` is the only primary object
- `workflow / script / gagent` are implementation kinds of the same member
- `service` is the member's published contract surface
- ordinary users should not have to type or manage `serviceId`

Issue `#325` tracks the backend feature work required to implement this decision.

## Decision

For Studio and team-member flows, the authoritative model is:

`scope -> member -> implementation -> published service -> endpoint -> run`

Concretely:

- `member` is the only primary Studio object
- every member owns exactly one stable `publishedServiceId`
- `publishedServiceId` is generated by backend at member creation time
- `publishedServiceId` is persisted and rename-safe
- `workflow / script / gagent` are implementation kinds, not parallel identity systems
- `Bind` means publishing the current member revision to that member-owned published service
- ordinary Studio users do not manually type `serviceId`

### Current Team Context Policy

- in the current cutover, Studio's visible `team` context is the selected `scope`
- `Team members` in Studio means "members in the currently selected scope-backed team context"
- backend does not need a separate first-class `team` authority to complete this member-first cutover
- if a first-class team model is added later, it must compose on top of the same member contract instead of replacing it

## Locked Rules

### 1. Identity

- `memberId` is the stable identity
- `memberId` is not `serviceId`
- `memberId` is not `workflowId`
- `memberId` is not `scriptId`
- `memberId` is not `actorId`

### 2. Published Service

- `publishedServiceId` is a contract surface, not a user-facing identity
- each member owns one stable `publishedServiceId`
- backend generates the value once and stores it directly on the member record
- rename must not regenerate or mutate `publishedServiceId`
- the generation rule must derive from immutable member identity, not from mutable display name

### 3. Implementation Kind

`Create member` must support exactly these implementation kinds:

- `workflow`
- `script`
- `gagent`

Those are implementation kinds of the same member lifecycle, not separate top-level object models.

### 4. Bind Semantics

`Bind` does not mean:

- choose an arbitrary existing scope service
- browse a service catalog as the primary task
- reuse the scope default service as the Studio semantic default

`Bind` means:

- resolve the selected member
- resolve that member's stable `publishedServiceId`
- publish the current implementation revision to that service
- return the invoke contract of that member-owned published service

### 5. Studio Routing

Studio route state must use real `memberId`.

If a Studio URL contains `memberId`, that value must identify a member, not a service.

Temporary compatibility for older service-backed links is allowed during migration, but the canonical route contract is:

- `memberId` means member identity
- `serviceId` means published contract identity

### 6. User-Facing Service ID Policy

For ordinary Studio flows:

- users do not type `serviceId`
- users do not pick an arbitrary service during member bind
- Studio shows `publishedServiceId` as read-only contract metadata

Advanced override behavior, if ever needed, is a separate design and not part of the default Studio member lifecycle.

## Required Backend Contract

Backend must provide a first-class member model with at least:

- `memberId`
- `scopeId`
- `teamId?`
- `displayName`
- `description?`
- `implementationKind`
- `implementationRef?`
- `publishedServiceId`
- `lifecycleStage`
- `createdAt`
- `updatedAt`

Backend must provide member-centric APIs for Studio:

- `POST /api/scopes/{scopeId}/members`
- `GET /api/scopes/{scopeId}/members`
- `GET /api/scopes/{scopeId}/members/{memberId}`
- `PUT /api/scopes/{scopeId}/members/{memberId}/binding`
- `GET /api/scopes/{scopeId}/members/{memberId}/binding`

The member binding path should reuse existing service binding capabilities internally, but Studio must not depend on the old scope-default binding contract as its primary model.

For the current Studio cutover, `scopeId` is the required container key for member APIs.
If a future first-class team API is added, it should remain an alternate access path to the same member authority, not a second member model.

## Consequences

- Studio left rail must become a real member roster
- `Create member` becomes true member creation, not asset creation masquerading as member creation
- `Build` becomes implementation editing for the selected member
- `Bind` becomes member publish, not service selection
- `Invoke` and `Observe` remain runtime surfaces, but their context is always the selected member
- existing scope-level runtime and service-level governance endpoints may remain during migration, but they are no longer the authoritative Studio semantic model

## Cutover Order

The required cutover order is:

1. Add backend member model and stable `publishedServiceId`
2. Add member-centric Studio APIs
3. Implement member binding by reusing existing service binding internals with explicit `ServiceId`
4. Switch Studio routing and left rail to real `memberId`
5. Remove the service-picker semantics from Studio Bind
6. Migrate legacy service-backed Studio links through a compatibility layer

## Non-Goals

This ADR does not define:

- the full team router redesign
- frontend visual layout details
- advanced manual service override UX
- immediate removal of all legacy scope-level runtime endpoints

## Outcome

After this ADR, Studio semantics are locked as:

- `member` is the primary object
- `workflow / script / gagent` are member implementation kinds
- `published service` is the member's stable contract surface
- `Bind` publishes a member revision to that member-owned stable service
